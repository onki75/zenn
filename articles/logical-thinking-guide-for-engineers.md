## はじめに

エンジニアとして技術力を磨くことはもちろん大切だが、それと同じくらい**論理的に話す力**が求められる場面は多い。設計レビュー、面接、チームへの提案、ドキュメント作成――すべてに共通するのは「自分の考えを構造的に伝える」というスキルだ。

特にAI駆動開発が普及した現在、エンジニアの主なアウトプットは「コード」から「指示・判断・レビュー」、つまり**言語**に変わりつつある。論理的に話す力は、もはやソフトスキルではなく**コアスキル**だ。

この記事では、論理的に話すための実践的な型と、よくある失敗パターン、そしてレベルアップのための考え方を紹介する。

## 基本の型: 結論→理由→具体例

論理的な説明の最小構成はこの3つだ。

```
結論:   〇〇である
理由:   なぜなら△△だから
具体例: 例えば□□
```

### 良い例

> **結論**: 子供に英会話を習わせるべきだ。
> **理由**: 英語を話せると将来の職業や居住地の選択肢が広がるから。
> **具体例**: 英語話者は世界に約15億人おり、英語ができるだけでコミュニケーション可能な相手が桁違いに増える。仮に日本が経済的な危機に陥ったとしても、海外で働くという選択肢が生まれる。

このように、結論→理由→具体例が**一本の線でつながっている**ことが重要だ。

## よくある失敗パターン

### パターン1: 循環論法

理由が結論の言い換えになっているケース。

```
❌ 視座を高く持つべきだ。なぜなら会社全体を見られるようになるから。
```

「視座を高く持つ」と「会社全体を見る」はほぼ同じ意味だ。これでは「なぜ？」に答えていない。

```
✅ 視座を高く持つべきだ。なぜなら面接官は「この人が入社後にどの範囲で貢献できるか」を見ているから。
```

**自分側の定義ではなく、相手側の視点で理由を説明する**と循環を避けられる。

### パターン2: 具体例のズレ

理由とは関係ない例を持ってくるケース。

```
❌ 結論: AIにコーディングを任せてよい
   理由: AIは人間の知能を超えているから
   具体例: 共通テストの模試で人間より高い成績を出した
```

コーディング能力の話をしているのに、共通テストの成績を例に出している。具体例は**理由を直接裏付けるもの**を選ぶ必要がある。

```
✅ 具体例: SWE-benchでは、AIが実際のGitHub issueを人間と同等以上の精度で解決できることが示されている。
```

### パターン3: 具体例ではなく対策を述べている

```
❌ ルールファイルのレビューが重要だ。
   例えば、Gitで差分を確認し、レビューを徹底することが大切だ。
```

これは「何をすべきか（対策）」であって「なぜ重要か（証明）」ではない。

```
✅ 例えば、ルールファイルに「テストは不要」と誤って記述された場合、
   以降のすべての実装でテストが省略される。
   1つの関数のバグは局所的だが、ルールファイルの誤りはプロジェクト全体に波及する。
```

具体例の役割は**「理由が正しいことを証明する」**ことだ。

### パターン4: 具体例が浅い

```
❌ AIは開発を高速化する。例えばテストを自動生成できる。
✅ AIは開発を高速化する。例えば、以前は手動でテストを書くのに1ファイル30分かかっていたが、
   AIに任せたら10ファイルを15分で生成できた。浮いた時間をテスト設計に使えるようになった。
```

**数字・前後比較・場面描写**を入れると、聞いた人が「本当にそうなんだ」と納得できる。

## 理由の抽象度を合わせる

理由の粒度は、結論と同じレベルにする。判定基準はシンプルだ。

```
理由を聞いた後に出てくる疑問が何か？

  「なぜ？」が出る   → 抽象すぎる（まだ理由が足りない）
  「例えば？」が出る → ちょうどいい ✅
  「だから何？」が出る → 具体的すぎる（理由ではなく具体例になっている）
```

### 実例

```
結論: 子供に英会話を習わせるべきだ

❌ 抽象すぎ:  将来のためになるから      → 「なぜ？」
✅ 適切:      職業や居住地の選択肢が広がるから → 「例えば？」
❌ 具体的すぎ: 英語話者が15億人いるから   → 「だから何？」
```

## 説得力を高める追加テクニック

基本型（結論→理由→具体例）ができたら、状況に応じて要素を追加する。

### 比較で説得する

「良い」と言うより「AよりBが良い、なぜならCだから」の方が強い。

```
❌ マルチエージェント開発は品質が高い。
✅ マルチエージェント開発は、単一AIで開発するよりも品質が高い。
   なぜなら単一AIでは不得意分野も含めてすべてを1つのモデルが引き受けるが、
   マルチエージェントなら各モデルの強みに合わせてタスクを配置できるからだ。
```

### 反論を先に潰す

相手が「でも〇〇では？」と思いそうなことを予測して先に対処する。

```
結論 → 理由 → 具体例 → 「たしかに〇〇という指摘はある。しかし△△」
```

重要なのは、**反論を認めつつも結論が成立する理由を示す**ことだ。反論に対して自分の結論を否定してしまうと、主張全体が崩れる。

### 前提・背景を加える

相手が文脈を知らない場合にだけ使う。

```
不要: 同僚に「この実装方針でいこう」と提案するとき
必要: 面接官に自分のプロジェクトを説明するとき
```

判断基準: **前提を省いたとき、相手が理由を理解できるか？**

できるなら省く。できないなら入れる。

## 全部盛りにしない

追加テクニックをすべて使う必要はない。

```
❌ 背景 + 前提 + 目的 + 結論 + 理由 + 比喩 + 比較 + 具体例 + 反論 + 言い換え
   → 聞いている側が疲れる

✅ 結論 + 理由 + 具体例 + 状況に応じて1〜2個追加
   → 伝わる
```

追加する要素は**「相手の疑問を先回りして消す」ために選ぶ**。自分が言いたいから入れるのではない。

## まとめ

論理的に話す力は、才能ではなく**型の習得と反復練習**で身につく。

1. **結論→理由→具体例**の型を常に意識する
2. 理由が結論の言い換えになっていないか確認する
3. 具体例は理由を証明するものを選び、数字や前後比較を入れる
4. 理由の抽象度は「聞いた後に『例えば？』と思えるレベル」に合わせる
5. 日常から「なぜ？」と疑問を持つ習慣をつける

これらを意識して話す練習を積み重ねれば、レビュー、提案、面接、あらゆる場面で伝わる説明ができるようになる。
